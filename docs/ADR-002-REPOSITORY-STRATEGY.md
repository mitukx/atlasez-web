# ADR-002: リポジトリ構成

- 状態: 採用
- 日付: 2026-07-22

## 決定

**案C（単一モノレポ）を簡素化した単一リポジトリ `atlasez-web`** を採用する。承認済みHTMLを人間が手動コピーする既存案（`atlas-chemistry-jpn` / `Edit_Chemistry-jpn` 方式）は**廃止**する。

```text
atlasez-web/
├── src/
│   ├── pages/            # 公式サイト(/)と学習サイト(/atlas/)のルート
│   ├── layouts/          # OrgLayout / AtlasLayout（2サイトのトーン分離）
│   ├── components/       # 共通UI
│   ├── content/          # ★ コンテンツ（執筆者が触るのはここだけ）
│   │   ├── concepts/     #   言語非依存の概念層 (YAML)
│   │   ├── articles/     #   言語別記事 (Markdown) ja/ en/ ...
│   │   ├── news/         #   お知らせ (Markdown)
│   │   ├── projects/     #   プロジェクト (YAML)
│   │   └── subjects/     #   分野・カテゴリ定義 (YAML)
│   ├── lib/              # グラフ・i18n・検索の抽象化
│   └── styles/           # デザイントークン
├── scripts/              # コンテンツ検証（循環・重複ID・リンク）
├── tests/                # Vitest / Playwright
├── docs/                 # 本ドキュメント群
└── .github/workflows/    # CI / デプロイ
```

## 三案の比較

| 観点                 | 案A: 分野×言語×公開/編集で分割       | 案B: コード/コンテンツ分離（3リポジトリ） | 案C: 単一モノレポ                                            |
| -------------------- | ------------------------------------ | ----------------------------------------- | ------------------------------------------------------------ |
| 執筆者の参加しやすさ | × リポジトリ選びから迷う             | ○ contentリポジトリだけ触る               | ◎ 1箇所・フォルダ明確                                        |
| 査読フロー           | × リポジトリ間の手動コピーで履歴断絶 | ○ PRベース                                | ◎ PRベース・プレビューで最終表示確認可                       |
| 誤公開の防止         | △ 手動コピーのミスが直接公開される   | ○                                         | ◎ `status: published`のみビルド＋CI必須チェック              |
| リポジトリ数         | × 23分野×言語×2で爆発                | ○ 3                                       | ◎ 1                                                          |
| 分野横断リンク       | × 別サイト間リンクで検証不能         | △ ビルド連携が必要                        | ◎ ビルド時に全リンク検証                                     |
| 多言語               | × 言語ごとに分裂                     | ○                                         | ◎ 概念層で束ねる                                             |
| ビルド時間           | ○ 個別は速いが全体像なし             | △ 2リポジトリのトリガー連携               | ○ Astroの差分ビルドで許容範囲。将来肥大化したら案Bへ分割可能 |

案Aは「誤公開の防止」を目的としていたが、実際には手動コピーこそが最大の誤公開リスク。案Bは記事数がビルド時間を圧迫する規模（数千記事）になった時の移行先として残す。

## 編集フロー（手動コピーの代替）

1. 執筆者が `src/content/articles/ja/...` にMarkdownを追加・編集（`status: draft`）
2. Pull Requestを作成（GitHub Web UIからでも可能。非エンジニアはブラウザ編集で完結）
3. CIが自動実行: スキーマ検証（Zod）／重複ID・リンク・循環検査／型チェック／ビルド／アクセシビリティ検査
4. 査読者がPRをレビューし、`reviewers` frontmatterに記名して承認
5. `status: published` に変更してmainへマージ
6. GitHub Actionsが自動ビルド・デプロイ。**`status: draft` の記事はビルドから除外**され、本番には公開済みのみが出る

下書きの確認はPRごとのプレビュービルド（アーティファクト/プレビュー環境）で行う。

## ブランチ運用

- `main`: 保護ブランチ。マージ＝本番公開。直接push禁止・CI必須・レビュー1名以上必須
- 作業ブランチ: `article/<id>`, `feature/<name>`, `fix/<name>`
